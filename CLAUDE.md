# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this repo is

An Astro static site, based on the official Astro blog starter, deployed to Cloudflare Workers. It has been
repurposed from a generic blog into a private family site ("Petrillo Family Activities & Planning") — the
homepage lists upcoming/archived family events and links out to standalone trip pages, rather than a blog feed.
The original blog demo content and scaffolding (`src/content/blog/`, `/blog`, `/rss.xml`) are still present and
functional but are no longer the site's focus.

## Commands

```bash
npm install               # install dependencies
npm run dev                # local dev server at localhost:4321
npm run build               # build production site to ./dist/
npm run preview             # build, then run it under wrangler (closer to prod: Workers runtime, bindings)
npm run check                # astro build && tsc && wrangler deploy --dry-run — full validation, no deploy
npm run deploy                 # wrangler deploy (ships ./dist to Cloudflare Workers)
npm run cf-typegen               # regenerate worker-configuration.d.ts from wrangler.json bindings
npm run astro -- <cmd>              # run any Astro CLI command, e.g. `npm run astro -- check`
```

There is no test suite and no linter/formatter configured in this repo — don't invent commands for either.
`npm run check` (build + `tsc` + dry-run deploy) is the closest thing to CI validation available and should be
run after non-trivial changes.

Node >= 22 is required (`engines` in package.json).

## Architecture

### Astro content collections vs. static `public/` micro-sites

There are two distinct ways content is served, and it matters which one you're editing:

1. **Astro-rendered pages** — driven by content collections defined in `src/content.config.ts`:
   - `blog`: Markdown/MDX in `src/content/blog/`, schema `{ title, description, pubDate, updatedDate?, heroImage? }`.
     Rendered by `src/layouts/BlogPost.astro` via `src/pages/blog/[...slug].astro`; listed at `src/pages/blog/index.astro`.
   - `events`: Markdown in `src/content/events/`, schema `{ title, description, date, location?, image? }`.
     Rendered by `src/layouts/EventPost.astro` via `src/pages/events/[...slug].astro`. An event's "Upcoming" vs
     "Archived" badge is derived at render time by comparing `date` to `Date.now()` — there's no manual status field.
   - `src/pages/index.astro` (the homepage) queries the `events` collection directly, splits it into upcoming
     (sorted ascending) and archived (sorted descending) lists, and renders both plus a hardcoded "Achieved
     Trips" section linking to the static trip pages below. New trips need a manually-added `<li>` here.

2. **Static HTML micro-sites under `public/`** — plain HTML/CSS/JS served as-is, completely outside Astro's
   routing, templating, and content collections. These are one-off pages, not scaffolding to generalize:
   - `public/trips/japan-2026/` — a photo-retrospective page. `photos.json` is a manifest of filenames in
     `./photos/`, regenerated by running `node generate-manifest.mjs` from that directory after adding images
     (filenames prefixed `tokyo-bay-`, `hakone-`, `osaka-`, or `tokyo-` get pinned to that section; everything
     else is spread evenly). See that directory's own `README.md` for the full workflow.
   - `public/upcoming-trip/` — a self-contained, single-file interactive trip planner (countdown timer, editable
     itinerary, packing checklist, budget table) with no build step. State is persisted client-side via
     `localStorage` only — there's no backend and no server-rendered personalization.

  When asked to add a new trip page, follow whichever existing pattern fits (photo retrospective vs. interactive
  planner) rather than inventing a third approach, and link it from the homepage's trip section.

### Everything else

- `src/consts.ts` holds site-wide `SITE_TITLE`/`SITE_DESCRIPTION`, imported anywhere via the `astro:content` /
  relative-import pattern already used in pages.
- `src/components/BaseHead.astro` centralizes `<head>` metadata (canonical URL, OpenGraph, Twitter cards) and
  is where `global.css` gets imported for every page — always include it in the `<head>` of new full-page layouts.
- `src/components/Header.astro` still contains the original Astro starter's nav links (Home/Blog/About) and
  Astro-project social icons (Mastodon/Twitter/GitHub for `@astro`) — these are stock template content, not
  Petrillo-family links; treat them as known leftovers rather than intentional design when reasoning about the nav.
- `src/pages/rss.xml.js` generates the RSS feed from the `blog` collection only (events are not included).
- Cloudflare integration: `astro.config.mjs` wires up the `@astrojs/cloudflare` adapter with `platformProxy`
  enabled (so `Astro.locals.runtime` / Worker bindings are available in `npm run dev`). `wrangler.json` points
  the Worker at `./dist` for both the built entrypoint and static assets. `worker-configuration.d.ts` is
  generated output (`npm run cf-typegen`) — don't hand-edit it, regenerate it instead if bindings change.
- `tsconfig.json` extends Astro's `strict` preset with `strictNullChecks` explicitly on.
